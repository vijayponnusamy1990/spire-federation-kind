# SPIRE Federation: Technical Deep Dive

This document provides a detailed technical overview of how SPIRE is installed, configured, and used for cross-cluster federation and authentication in this project.

## 1. SPIRE Infrastructure Installation

The SPIRE infrastructure is deployed using Helm charts located in `helm/spire`.

### SPIRE Server

- **Deployment Type**: `StatefulSet` (to maintain the SQLite database and keys on disk).
- **Service**: Exposed via a `LoadBalancer` (MetalLB) on port `8443` for federation.
- **Node Attestation**: Uses the `k8s_sat` (Kubernetes Service Account Token) attestor. The server validates the SPIRE Agent's identity by verifying its service account token against the Kubernetes Token Review API.

### SPIRE Agent

- **Deployment Type**: `DaemonSet` (runs on every node).
- **Workload Attestation**: Uses the `k8s` workload attestor. It identifies workloads based on their Kubernetes namespace, service account, and pod labels.
- **Communication**: Workloads connect to the SPIRE Agent via a Unix Domain Socket mounted at `/run/spire/sockets/agent.sock`.

## 2. Workload Registration

Registration is the process of telling SPIRE which workloads are allowed to receive which SPIFFE IDs.

### The Registration Flow (`3-register.sh`)

1. **Agent Identification**: The script first identifies the SPIFFE ID of the running SPIRE Agent. In this setup, the Agent ID is dynamically generated by the `k8s_sat` attestor.
2. **Entry Creation**: The script uses the `spire-server entry create` command to map a set of selectors to a SPIFFE ID.
    - **Selectors**: `k8s:sa:<service-account>` is used to identify the workload.
    - **Parent ID**: The entry is mapped to the Agent's SPIFFE ID.
    - **Federation Flag**: The `-federatesWith` flag is added to allow the workload to trust the foreign trust domain.

**Example Entry:**

- **SPIFFE ID**: `spiffe://cluster1.com/server`
- **Selectors**: `k8s:sa:server-service-account`
- **Federates With**: `cluster2.com`

## 3. Trust Domain Federation

Federation allows workloads in different trust domains (clusters) to verify each other's identities.

### Bundle Exchange (`2-bootstrap.sh`)

For federation to work, SPIRE Servers must exchange their public signing certificates (Trust Bundles).

- **Exporting**: `spire-server bundle show -format spiffe` prints the local trust bundle.
- **Importing**: `spire-server bundle set -id spiffe://<foreign-domain>` saves the foreign bundle into the local SPIRE database.

### Configuration (`server-configmap.tpl`)

The SPIRE Server is configured with a `federation` block that defines the HTTPS endpoint (exposed via MetalLB) where it can fetch bundle updates from the federated partner.

## 4. Authentication Flow (MTLS)

Workloads use the `go-spiffe` library to automatically handle certificate rotation and MTLS handshakes.

### Backend (`stock-quotes-service`)

The backend creates an MTLS server using:

```go
tlsconfig.MTLSServerConfig(x509Src, bundleSrc, tlsconfig.AuthorizeAny())
```

- **x509Src**: Provides the server's own certificate (SVID).
- **bundleSrc**: Provides the trust bundles for both the local domain and the federated domain.

### Frontend (`broker-webapp`)

The frontend creates an MTLS client. Because it dials the backend via a direct IP address (which is not present in the SPIFFE ID's SANs), we use a custom configuration:

```go
tlsConfig := tlsconfig.MTLSClientConfig(x509Src, bundleSrc, authorizer)
tlsConfig.InsecureSkipVerify = true // Skip standard hostname/IP validation
```

**Why `InsecureSkipVerify`?**
Standard Go TLS validation expects the IP address `172.18.x.x` to be in the certificate. SPIFFE certificates only contain the SPIFFE ID in the URI SAN. By setting `InsecureSkipVerify`, we skip the hostname check but **still maintain full security**, as the `go-spiffe` library still validates the certificate chain against the SPIRE trust bundles and verifies the SPIFFE ID via the `authorizer`.

## 5. SVID Types: X.509 (mTLS) vs. JWT

SPIRE can issue two types of SPIFFE Verifiable Identity Documents (SVIDs). This project exclusively uses **X.509 SVIDs**.

### X.509 SVIDs (Used Here)

- **Mechanism**: Standard X.509 certificates.
- **Authentication**: Mutual TLS (mTLS).
- **Use Case**: Best for service-to-service communication where a persistent connection or high-performance handshake is needed.
- **Why we chose it**:
  - It provides **encryption** in transit automatically.
  - It supports **mutual authentication** (both client and server verify each other).
  - SPIRE handles the rotation of these certificates transparently.

### JWT SVIDs

- **Mechanism**: JSON Web Tokens.
- **Authentication**: Bearer tokens in HTTP headers (e.g., `Authorization: Bearer <token>`).
- **Use Case**: Best for authenticating with services that don't support mTLS (like some Cloud APIs) or for passing identity through multiple architectural layers (like a frontend to a legacy backend).
- **Limitation**: Unlike X.509, JWTs do not provide transport-layer encryption on their own; they must be sent over an existing secure channel (like standard TLS).

## 5. Troubleshooting the State

### Persistent Data Pitfall

SPIRE Server and Agent use `hostPath` volumes for data storage. If you delete a pod without wiping the host directory (`/var/spire-data`), the new instance will try to reuse old, potentially expired or mismatched data.

**Clean Reset Command:**

```bash
docker exec <node-name> rm -rf /var/spire-data /run/spire/sockets
```

## 6. Hub-Spoke Architecture Extension

This demo supports an advanced Hub-Spoke model where a central SPIRE Server (simulated as a Docker container on the `kind` network) acts as a high-level trust anchor.

### Components

- **Hub**: `hub.com` (Docker container IP 172.18.0.x).
- **Spokes**: `cluster1.com` and `cluster2.com`.

### Federation Mesh

In this model, spokes federate with the Hub, and the Hub federates with all spokes. This creates a centralized trust graph that can be extended to hundreds of clusters or VMs.

### MTLS in Hub-Spoke

Even with a central Hub, workloads in different spokes still perform direct MTLS. Since they are federated with the Hub, they can also trust identities issued by the Hub (e.g., a "control-plane" service running in the HUB VM).
